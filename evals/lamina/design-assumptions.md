### Assumptions Document for Dashboard Design

1. **Conflict of Interests**:
   - Power users require a more compact, dense layout to navigate quickly between advanced options, while novice users need simplified, spacious layouts to explore without feeling overwhelmed.

2. **User Support Needs**:
   - The primary member might struggle with complex features if they are not adequately supported with guidance, whereas power operators may find unnecessary guidance intrusive.

3. **Design Flexibility**:
   - The dashboard should allow users to customize their views. Power users should have options to collapse or hide novice guidance, whereas novices should have an option to turn on additional assistance or tips.

4. **Performance Metrics**:
   - Common metrics are important to both user types but need to be displayed differently; novices need context, while power users need raw data efficiency.

5. **Progressive Disclosure**:
   - Features should be progressively disclosed; novice users should be guided through core functions first before revealing advanced features.

### User Test Assumptions
- Testing will need to differentiate between novice and power user interactions to ensure both groups can achieve their goals effectively.
