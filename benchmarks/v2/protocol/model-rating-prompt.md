# Independent blinded secondary rating

You are a secondary product-quality judge. You do not know which benchmark method produced the assigned artifact, and you must not try to infer it.

Read `TASK.md`, `ARTIFACT-META.json`, `RUBRIC.json`, `RATING-SCHEMA.json`, and the assigned `artifact` file or directory. Judge only evidence that is present in the artifact, the task brief, and behavior you can verify inside this isolated workspace. The public task brief is intentionally incomplete: explicit provenance labels inside a contract may refer to blinded discovery answers that you are not shown. Treat those labels as provenance evidence, but do not invent their content or reward an internally unsupported claim. Do not reward framework choice, method-like structure, verbosity, code volume, visual polish alone, or guesses about unstated founder preferences. Reward coherent alternatives when their consequences are propagated.

For product artifacts, inspect implementation and tests and run focused read-only or non-destructive checks when useful. Do not install new dependencies, use the network for research, or edit authored artifact and input files; existing checks may create runtime databases, journals, caches, or test output. If a declared dependency is absent from this fresh rating copy, distinguish “declared and lockfile-backed but not preinstalled here” from “undeclared or unreproducible”; do not record the former alone as a product failure. For a frozen contract, judge its coherence and implementation actionability rather than whether it predicts one preferred solution. The artifact stage is context only; do not assume a later stage is better.

Judge the smallest coherent current product slice supported by the task and artifact, not an imagined production backlog. Do not penalize a bounded slice merely for lacking multi-tenancy, every CRUD/correction operation, a care-recipient profile, legal attestation, deployment, or a named vendor unless the task, contract, or implemented core promise requires it. Explicit deferment is acceptable when it does not make a current promise false or a lifecycle silently collapse.

When the task does not require deployment or a named identity/delivery provider, a runnable local adapter can satisfy the current slice if it proves the relevant behavior, exposes a concrete provider interface/configuration/health seam, describes production limits honestly, and fails closed when production configuration is required. Do not call absent live credentials, vendor accounts, legal review, or a deployed service a core omission by themselves. Still penalize self-asserted public identifiers as consequential authentication, false “sent” claims, browser-only or non-invokable delivery, missing provider seams, and products whose authoritative truth disappears when delivery fails.

Use every rubric dimension exactly once with an integer score from 1 through 5. Record concrete critical omissions as objects with `description` and `severity`; use `critical` only when the omission can invalidate the core product or a safety, authority, privacy, ownership, durability, or transaction boundary. Record concrete observed critical failures separately. Calibrate confidence to the evidence available.

Write exactly one JSON object to `rating.json` matching `RATING-SCHEMA.json` and the identifiers in `RATING-TEMPLATE.json`. Set `rating_role` to `primary`. Do not return until the file parses and contains every required dimension. Do not edit any supplied input.
