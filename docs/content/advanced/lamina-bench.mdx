# LaminaBench

LaminaBench is a public product-behavior implementation benchmark. It compares the real Lamina workflow (init → design → implement → verify → fix) against an equally scaffolded generic plan→review loop on the same product briefs.

## Quick path

LaminaBench is for evaluators and researchers, not end users running Lamina day to day. To try Lamina yourself, start with [Quickstart](/getting-started/quickstart).

## What it measures

| System | Measures |
|--------|----------|
| **LaminaBench** (`benchmarks/`) | Product behavior outcomes — does the implemented code match domain rules? |
| **Skill evals** (`evals/`) | Skill compliance — does Lamina follow its own workflow rules? |

LaminaBench answers: "Does the Lamina design→verify loop produce better product behavior than a matched generic loop?"

## Methodology (Design C)

Ecological matched-phase comparison:

| Arm | Five-phase workflow |
|-----|---------------------|
| **Control** | plan → build order → implement → review → fix |
| **Treatment** | `/lamina-init` → `/lamina-design` → implement → `/lamina-verify` → fix |

Both arms receive the same `instruction.md`, the same phase count, and equal per-phase turn budgets. The treatment arm has Lamina skills and workflow conventions. Planning artifacts are excluded from scoring; only implemented application source is judged.

Scoring uses checklist coverage and LLM rubric on implemented source via Harbor Rewardkit.

<Callout type="info">
**Why this design?** LaminaBench primary methodology is Design C (matched 5-phase harness). We rejected asymmetric phased treatment vs single-shot control, `/lamina`-prefix-only comparison, and demoted SkillsBench single-session skills-on/off to an optional ablation. See [Design evolution](#design-evolution) below.
</Callout>

<Callout type="info">
**Advanced:** v1 status — no live published results yet. The benchmark infrastructure is in place; results will be published when replication runs complete.
</Callout>

## Corpus (v1)

10 tasks × 2 arms (control + treatment) = 20 trials per run. Tasks cover realistic product scenarios with golden checklists for scoring.

## Design evolution

LaminaBench went through several design iterations before landing on Design C.

| Design | Status | Why |
|--------|--------|-----|
| **A — Asymmetric phased** (5-phase treatment vs 1-shot control) | Rejected | Confounds Lamina with extra process; not fair |
| **B — SkillsBench single session** (skills on/off, one rollout) | Optional ablation | Does not match real slash-command usage |
| **Prefix — `/lamina <brief>`** | Rejected | Incomplete loop; confounds prefix + skills |
| **C — Matched phases** (5-phase Lamina vs 5-phase generic) | **Primary** | Real usage + equal control scaffolding |

Full rationale: [benchmarks/METHODOLOGY.md — Design evolution](https://github.com/aryaniyaps/lamina/blob/main/benchmarks/METHODOLOGY.md#design-evolution--what-we-considered-and-why-we-landed-here).

## Running the benchmark

Prerequisites:

- Node.js 20+
- Docker
- Anthropic API credentials

```bash
npm install
npm run bench:env-check
npm run bench:validate
npm run bench:harbor:sync -- --tasks task001
npm run bench:all
```

Key scripts:

| Script | Purpose |
|--------|---------|
| `npm run bench:env-check` | Verify credentials and Docker |
| `npm run bench:harbor:sync` | Sync Harbor task workspaces |
| `npm run bench:run` | Run matched phased trials |
| `npm run bench:aggregate` | Paired metrics → `results/aggregated/benchmark.json` |
