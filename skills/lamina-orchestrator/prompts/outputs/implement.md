# Ship pack: <target>

## Stack

Stack-agnostic. The coding agent chooses language, framework, UI toolkit, and persistence that fit the brief and repo.

State any **hard constraints from the brief** (e.g. mobile-first, offline-capable, must run locally). Do **not** prescribe Auth0, Firebase, AWS, CI vendors, or a default framework.

## Module / file map

Where domain rules, auth/permissions, primary workflows, and seed/fixture data should live — as a logical map the implementer can map onto their stack.

The map must describe a **behavioral core**, not only types and screens: mutable product state/persistence, invariant enforcement, actor-scoped selectors/authorization, workflow transitions, and scenario/recovery handling. Put consequential permissions, privacy, money/inventory constraints, and shared-state invariants behind a trusted service/store boundary; view filtering alone is not enforcement. A context containing only an id, type-only domain files, no-op event handlers, and placeholder comments is not an implementation architecture.

## Run locally

≤3 commands (or equivalent) to install, **load seed**, and run the product for verify. Adapt to the chosen stack; do not invent cloud deploy steps.

## Reachability graph (first-class)

From `domain.dependencies[]` + `workflows[].requires` / `provides` / `standalone`:

1. **Build order** — topological (setup/provides workflows before dependents).
2. **Per edge table:**

| dependency id | from | requires | mode | unmet behavior | scenario_ref | acceptance (copy) |
|---------------|------|----------|------|----------------|--------------|-------------------|
| … | workflow.… | entity.… | unreachable\|degraded\|blocked_ui\|recover | … | … | … |

3. For `degraded`: list `degraded_surfaces` and recovery. For `blocked_ui`/`recover`: name recovery workflow/screen.
4. **Never** silent happy-path when a dependency is unmet.

## Invariants → enforcement

For each `domain.invariants[]` id: where it must hold (data constraint and/or request/handler check) and the **rejection shape** when violated. Match the brief’s channel: if `screens[]` with `status: new` exist for that flow, name a UI (or UI+API) rejection surface; if the contract is API/headless only, an API rejection shape is enough.

## Permissions matrix

Actor × operation × **resource_filter** (from `actors[].resource_filters`). Ownership fields used in filters must exist on entities. Forbidden operations → `scenarios[]` category `permission`.

## Workflows → surfaces

For each `workflows[]` id: steps, linked `screens[]`, **success / degraded / failure** outcomes (align with dependency modes). Include every `screens[]` with `status: new`. Add an executable-chain row:

| workflow | reachable trigger | trusted enforcement | mutation + persistence | actor-scoped projection | visible outcome/recovery |
|----------|-------------------|---------------------|------------------------|-------------------------|--------------------------|
| … | … | … | … | … | … |

For each cross-actor handoff, include a named delivery boundary, recipient open/accept/reject, authenticated recipient binding, trusted handoff-state validation, durable relationship or permission change, both projections, and expiry/replay recovery. Generating a sender-only artifact, exposing a handoff credential only in verification mode, or claiming a pre-privileged seed session does not complete the row.

For local/demo implementations, a real in-process service + durable store can be sufficient; hard-coded seed display or a UI reducer that any actor can bypass is not.

For unavailable/out-of-scope third-party systems, add a **production-shaped adapter** row: named port/interface, begin operation, state/nonce, callback/exchange, durable attempt/outcome, retry, and replaceable local fake. The primary UI invokes the same adapter path it would use with a real provider; do not expose fixture outcome choices as the product workflow. Put actor/session resolution in a named trusted module, reject or explicitly onboard/sign in requests without credentials, and isolate actor/scenario selectors behind explicit verification mode.

Required edge cases must be generated by ordinary product paths, not only fixtures: normal inputs create conflict/reconciliation records, handoffs create recipient-reachable delivery records, and external failures create retryable attempts. Specify serialization/version checks for shared mutations, refresh all affected actor/offline projections, and either encrypt sensitive offline data behind a real key boundary or label/minimize it honestly.

Identity establishment must prove control even in a local adapter (password, passkey, or delivered one-time challenge); accepting a public identifier alone is not authentication. Add a normal creation/configuration path for every brief-critical record, relationship, and lifecycle variant advertised in the UI. Validate cross-record relationships and publish in-memory state only after the durable commit succeeds; include a failure-injection probe. Enumerate every consequential mutation against every disallowed role, including actor-scoped status, acknowledgment, completion, and preference operations.

## Scenarios → acceptance

For each `scenarios[]` id: copy `acceptance` **verbatim** from `run.yaml`. Include every `dependency_ref` scenario. No “handle gracefully.”

## Seed

Copy `seed.summary` and fixture bullets — the world verify/persona walks assume.

## Must-implement checklist

Generate from the contract — **every row must be realized in application source** (not comments alone). Tick while coding:

1. **Entities and state** — one checkbox per required `domain.entities[].id`, including its relationships and mutable state path; a TypeScript interface alone is not complete.
2. **Invariants and permissions** — one checkbox per `domain.invariants[].id` and actor resource filter, naming executable enforcement and rejection/filter behavior.
3. **Screens and workflows** — one checkbox per `screens[].id` with `status: new` and each `workflows[].id` (cite the app path and the complete trigger → enforcement → mutation/persistence → projection → feedback chain).
4. **Scenarios** — one checkbox per `scenarios[].id` with its `acceptance` copied verbatim and the state/trigger used to exercise it.
5. **Forbidden content** — one checkbox per `forbidden_content[]` entry: name the **rejection / absence surface** matching the brief channel (UI and/or API as contracted — not “API error” by default when screens exist).
6. **A11y** — for each `screens[]` with `status: new`, emit separate screen-qualified checkboxes for `a11y.labels`, `a11y.touch_min_px` (primary controls ≥ that px), `a11y.color_not_only`, navigation/rerender focus, invalid-field focus/association, async live announcements, busy/disabled state, and semantic progress/status; never use one shared a11y checkbox for all screens.
7. **Tradeoffs** — for each `tradeoffs[].id`, ship the `choice` and the mitigating surface named in `cost` / `surfaces`; preserve the id derived from the current run rather than substituting evaluator-oriented wording.

Every checkbox must end with **expected evidence** (`Expected evidence: <planned path/symbol>`). This is a read-only target for the external coding agent; implementation must not rewrite the ship pack. Do not treat a comment, import, route registration, type declaration, or nonexistent path as satisfying that target.

## Done-when

Union of Must-implement ids: `dependency.*`, `workflow.*`, `scenario.*`, `invariant.*`, `screen.*`, `forbidden.*`, `a11y.*`, `tradeoff.*`.

## Out of scope

Copy `out_of_scope[]` and `forbidden_content[]` from `run.yaml`. Plus default ops bans unless the brief requires them:

- CI/CD, deploy scripts, monitoring/APM
- Push/notification vendor wiring
- Real bank OAuth / production IdP unless required

## When implementing from this file

1. **Read first** — open this `implement.md` and the matching `run.yaml` before writing application source. Do not invent a narrower scope from memory.
2. **One session, full checklist** — realize every Must-implement checkbox in this coding session. If `screens[]` with `status: new` are listed, ship those surfaces in app source now (routes/components/templates). Do **not** ship API/domain only and defer contracted screens to a “later frontend” pass.
3. **Done means checklist done** — do not claim complete while any Must-implement id remains unrealized in source.
4. **Scaffold ≠ shipped** — `package.json`, `tsconfig`, shared type stubs, or empty module shells do **not** satisfy Must-implement. Do not say checklist items are “represented,” “scaffolded,” or “ready for a coding session” — **this turn is that coding session**. Keep writing application source until contracted behavior (handlers/routes/UI as required by the checklist) exists on disk outside `.lamina/`.
5. **Buildable before done** — every module imported by entrypoints/routers/screens must exist on disk. Do not claim complete while any imported path is missing (e.g. `Settings` screen referenced but not written). Prefer a quick compile or import-path check before ending.
6. **Brief-named surfaces** — ship every primary flow and secondary surface named by the current task brief when it appears on the Must-implement checklist; do not omit it as “phase 2.”
7. **Do not burn this turn on `.lamina/`** — if `implement.md` / `ready_to_build` are already present, do not regenerate the ship pack; implement the product. If they are missing, finish design artifacts only long enough to unblock, then immediately continue into application source in the same turn.
8. **Behavior before breadth claims** — build the smallest coherent vertical core first: seeded/durable state → trusted invariant/permission enforcement → workflow transitions/recovery → actor-scoped reads → screens bound to that behavior. Then complete the remaining checklist. A screen whose button has a no-op/placeholder handler or whose data is hard-coded independently of product state is still unticked.
9. **Probeable states** — expose deterministic local seed/reset controls or documented fixture commands for each critical actor, denial, empty, stale/offline, conflict, and recovery state. Do not make verification depend on reading test files or editing source constants.
10. **Independent buildability** — declare a clean build/compile/typecheck script discoverable by the harness (for Node, `package.json#scripts.build`) and run it before completion. `start`, smoke scripts, and tests remain separate supplemental evidence.
10. **Evidence-based finish** — before claiming completion, inspect every checklist evidence path, search for placeholder/TODO/no-op handlers, run the actual typecheck/build, and verify imported paths exist. A script name is not a pass; inspect its exit status. If time remains insufficient, report the exact unticked rows; never convert partial work into a completion claim.
